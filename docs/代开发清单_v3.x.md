## 二、任务拆解

### 1. 配置中心 MCP 适配（Apollo / Nacos）

**目标**：将 Apollo、Nacos 等配置中心适配为 MCP 协议服务，供 AI 在分析时获取配置快照、命名空间、配置变更等证据。

#### 1.1 Apollo MCP Server

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 1.1.1 Apollo API 调研 | 梳理 Apollo Open API（配置查询、命名空间、发布历史） | 调研文档 |
| 1.1.2 Apollo MCP Server 实现 | 新建 `mcp-servers/apollo`，实现 tools：`config.get_namespaces`、`config.get_config`、`config.get_release_history` | 可运行 MCP Server |
| 1.1.3 认证与鉴权 | 支持 AppId + 环境 + Token / 私钥；支持私有化部署地址配置 | 配置项、README |
| 1.1.4 集成测试 | 对接 RootSeeker MCP 网关，验证工具可被 AI 调用 | 测试用例 |

#### 1.2 Nacos MCP Server

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 1.2.1 Nacos API 调研 | 梳理 Nacos 配置 API（配置列表、历史、监听） | 调研文档 |
| 1.2.2 Nacos MCP Server 实现 | 新建 `mcp-servers/nacos`，实现 tools：`config.list`、`config.get`、`config.history` | 可运行 MCP Server |
| 1.2.3 认证与多环境 | 支持用户名/密码、命名空间；支持 Nacos 2.x | 配置项、README |
| 1.2.4 集成测试 | 对接 RootSeeker，验证工具调用 | 测试用例 |

#### 1.3 统一配置与文档

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 1.3.1 配置模板 | `config.example.yaml` 增加 Apollo/Nacos MCP 配置示例 | 配置片段 |
| 1.3.2 部署文档 | 编写 Apollo/Nacos MCP 部署与接入说明 | docs/ |

### 2. 基础设施监控 MCP（Linux / K8s / Docker）

**目标**：适配 Linux、K8s、Docker 的监控采集，支持 CPU、内存、栈信息等证据收集，形成 MCP 协议服务。

#### 2.1 Linux 主机监控

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 2.1.1 指标采集设计 | 定义需采集指标：CPU、内存、磁盘、网络、进程列表、栈信息（jstack/gdb） | 设计文档 |
| 2.1.2 Linux MCP Server | 实现 tools：`host.cpu`、`host.memory`、`host.process_list`、`host.stack_dump`（需 SSH/Agent） | MCP Server |
| 2.1.3 安全与权限 | 白名单主机、密钥管理、只读权限约束 | 安全设计 |

#### 2.2 Kubernetes 监控

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 2.2.1 K8s API 调研 | 梳理 metrics-server、kubectl top、Pod 日志、事件、describe 等 | 调研文档 |
| 2.2.2 K8s MCP Server | 实现 tools：`k8s.pod_metrics`、`k8s.pod_logs`、`k8s.events`、`k8s.describe_pod` | MCP Server |
| 2.2.3 认证方式 | 支持 kubeconfig、ServiceAccount、内网集群 | 配置项 |

#### 2.3 Docker 监控

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 2.3.1 Docker API 调研 | 容器 stats、logs、inspect | 调研文档 |
| 2.3.2 Docker MCP Server | 实现 tools：`docker.stats`、`docker.logs`、`docker.inspect` | MCP Server |
| 2.3.3 与 K8s 整合 | 可选：Docker 作为 K8s 底层时的去重与关联 | 设计说明 |

#### 2.4 栈信息专项

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 2.4.1 Java 栈采集 | jstack / arthas 集成，按 PID 或容器获取线程栈 | tool: `stack.java_dump` |
| 2.4.2 通用栈采集 | 支持 gdb、py-spy 等，按语言/运行时选择 | 扩展接口 |

### 4. 数据库 MCP 与 SQL 智能生成

**目标**：对接 MySQL、SQL Server 等常见数据库，形成 MCP 服务，支持 AI 生成查询 SQL，获取实时数据，保障链路选择与错误定位准确。

#### 4.1 数据库连接与安全

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 4.1.1 连接池与配置 | 支持多数据源、只读账号、连接超时、SSL | 配置模型 |
| 4.1.2 权限与审计 | 仅允许 SELECT；记录执行 SQL、影响行数；防注入 | 安全设计 |

#### 4.2 MySQL MCP Server

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 4.2.1 Schema 发现 | `db.schema`：库、表、字段、索引 | tool |
| 4.2.2 SQL 执行 | `db.execute`：执行只读 SQL，限制行数/超时 | tool |
| 4.2.3 智能 SQL 生成 | 结合错误上下文（表名、字段、trace_id）生成候选 SQL，供 AI 选择或微调 | 提示词 + 工具 |

#### 4.3 SQL Server MCP Server

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 4.3.1 驱动与连接 | pyodbc / pymssql，支持 Windows 认证、SQL 认证 | 实现 |
| 4.3.2 工具对齐 | `db.schema`、`db.execute` 与 MySQL 接口一致 | 实现 |

#### 4.4 AI 链路选择支持

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 4.4.1 上下文注入 | 从 error_log 解析表名、业务 ID、时间范围，注入 Plan/Synthesize | 提示词 |
| 4.4.2 SQL 生成 Prompt | 提供「表结构 + 错误上下文 → 生成 SELECT」的 few-shot 示例 | prompts.py |
| 4.4.3 结果校验 | 对空结果、超时给出明确提示，引导 AI 调整查询 | 工具逻辑 |

### 8. 场景化编排（批量报错关键根因定位）

**目标**：支持一批报错集合的场景，从中识别关键报错，进行场景化编排，避免逐个分析的低效。

#### 8.1 批量输入与去重

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 8.1.1 批量 Ingest API | `/ingest/batch` 接收多条错误日志，返回 batch_id | API |
| 8.1.2 聚类与去重 | 按 trace_id、堆栈特征、错误类型聚类，合并相似报错 | 算法 + 实现 |
| 8.1.3 关键报错排序 | 按时间、频率、堆栈深度、业务影响等排序，选出「关键报错」 | 策略 |

#### 8.2 场景化编排流程

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 8.2.1 编排模式设计 | 定义「批量模式」：先聚类 → 选关键 → 对关键报错做全量分析 → 可选对关联报错做轻量分析 | 设计文档 |
| 8.2.2 Plan 扩展 | Plan 阶段支持「本批报错列表 + 关键报错 ID」上下文 | 提示词 |
| 8.2.3 证据共享 | 同一 batch 内，证据可跨报错复用（如 correlation 一次拉取多条） | 实现 |

#### 8.3 报告与展示

| 子任务 | 说明 | 产出 |
|--------|------|------|
| 8.3.1 批量报告结构 | 根因报告 + 关联报错列表 + 聚类说明 | 领域模型 |
| 8.3.2 Admin 展示 | 批量任务详情、关键报错高亮、关联关系图 | 前端 |

## 三、实施建议

### 3.1 推荐排期（按依赖与价值）

```
Phase 1（基础能力）
├── 4. 数据库 MCP（证据链关键）
├── 1. 配置中心 MCP（配置漂移常见根因）
└── 6. Admin 系统（运维与观测基础）

Phase 2（监控与云）
├── 2. 基础设施监控 MCP
├── 3. 云服务商对接
└── 7. 构建工具只读对接

Phase 3（场景与体验）
├── 8. 场景化编排
└── 5. 版本工具扩展（SVN）
```

### 3.2 风险与依赖

| 风险 | 缓解 |
|------|------|
| 各云 API 差异大 | 先做阿里云，抽象通用接口，再扩展腾讯/AWS |
| 数据库只读与安全 | 严格白名单、审计日志、只读账号 |
| 批量编排复杂度 | 先实现「选关键报错 + 单条全量分析」，再迭代聚类与证据共享 |

### 3.3 验收标准（每模块）

- [ ] MCP Server 可独立启动，tools 可被 RootSeeker 网关发现并调用
- [ ] 配置项完整，文档可复现部署
- [ ] 至少 1 个端到端分析用例验证证据收集有效
- [ ] 安全评审通过（权限、注入、敏感信息脱敏）
