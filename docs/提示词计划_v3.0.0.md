# RootSeeker v3.0.0 提示词计划

本文件是一组面向 IDE/Agent 的提示词（Prompt Pack），用于驱动实现 v3.0.0。  
v3.0.0 在 v2.0.0 基础上新增两大重点：

> **Cline 参考**：工具错误与参数修正相关描述已基于 [cline-3.72.0](https://github.com/cline-dev/cline) 源码核对；本地可参考 `../cline-3.72.0/` 目录。

- 外部依赖识别升级：支持 Java（Maven/Gradle）、Python（requirements/pyproject/pip freeze）等依赖识别与版本解析。
- AI 代码解析能力升级：参考 Cursor/Trae/Cline 的 IDE 上下文管理策略，优化“先勘探、再定位、再综合”的分析质量。

## 0) 先分析：现状与对比结论

### 0.1 RootSeeker 当前能力（已实现）

- 服务级依赖：通过 `ServiceGraphBuilder` 扫描代码与配置，提取 URL host 与 `@FeignClient` 等线索构建服务调用关系。
- 方法级依赖：通过 `CallGraphExpander` 在 Java/Python 上做 Tree-sitter/回退解析。
- 上下文管理：主流程已是 Plan -> Act -> Synthesize -> Check，多轮迭代并具备上下文压缩、证据缓存、重复读取优化。
- 外部依赖识别：当前以“读取构建文件作为证据”为主（如 `pom.xml`、`build.gradle`、`requirements.txt`），尚未形成统一结构化依赖画像与动态解析能力。
- 依赖代码识别：当前不接入 Java LSP（如 JDT LS），也不解析/索引 `~/.m2` 中的 jar 或 sources.jar；因此对“依赖库源码”的定位能力弱，主要依赖错误堆栈字符串与仓库内代码来推断。

### 0.2 Cursor / Trae / Cline 的可借鉴能力

- Cursor 风格：工作区语义检索 + 符号级定位 + 按需注入上下文，避免一次性塞满上下文窗口。
- Trae 风格：显式任务分解与阶段化上下文管理，强调可追踪的计划与执行链路。
- Cline 风格：勘探优先、工具结果回灌、上下文压缩与规则注入，必要时通过命令执行获取更准确运行态信息。
- **Cline 工具错误处理**（参考 cline-3.72.0 源码）：工具结果直接回写 conversation，主 LLM 在下一轮自行修正；渐进式错误仅针对 write_to_file 的 content 缺失（1/2/3+ 次分级）；缺参时 `missingToolParameterError(paramName)` 明确标出参数名；mistake_limit 达上限时 ask 用户 feedback 后继续。

### 0.3 差距与优化空间

- 依赖识别仍偏“静态文本证据”，缺少“静态解析 + 动态命令解析”的双轨策略。
- 上下文压缩主要是字符维度，缺少 token 预算与证据相关性重排机制。
- 多语言代码解析能力不均衡，Java/Python 较强，其他语言需要统一兜底策略。

## 1) v3.0.0 目标摘要

1. 外部依赖识别能力升级  
   - 覆盖 Maven/Gradle/Python 生态，形成结构化依赖视图（直接依赖、传递依赖、版本冲突风险）。
2. MCP 可控执行能力升级  
   - 通过 MCP 安全执行依赖分析命令，禁止任意 Shell 注入。
3. AI 代码解析能力升级  
   - 参考 Cursor/Trae/Cline 的上下文管理能力，提升定位准确性、可解释性与稳定性。

## 2) v3.0.0 总控 Prompt（项目级规则）

```text
【身份】你是 RootSeeker v3.0.0 的错误分析与依赖识别编排器，按“先分析现状 -> 再设计方案 -> 再落地实现 -> 最后验证”的顺序执行。

【目标】
1) 增强外部依赖识别：支持 Java(Maven/Gradle)、Python(requirements/pyproject/pip freeze)；
2) 增强 AI 代码解析能力：参考 Cursor/Trae/Cline 的上下文管理 IDE 能力；
3) 在 MCP 框架内提供安全、可控、可回退的依赖分析工具执行能力。

【核心策略】
- 依赖识别采用“双轨”：
  - 轨道A：静态解析（快）— 解析 pom.xml/build.gradle/requirements.txt/pyproject.toml；
  - 轨道B：动态解析（准）— 在必要时执行 maven/gradle/pip 只读命令获取解析后的依赖树。
- AI 代码解析采用“分层上下文”：
  - 任务层：问题目标、轮次决策；
  - 证据层：工具结果、可追溯参数；
  - 代码层：符号/调用链/文件片段，按需加载。

【约束】
- 不允许直接执行未白名单命令；所有命令执行必须通过 MCP 工具封装。
- 未安装构建工具时必须降级到静态解析并给出明确说明。
- 结论必须绑定证据来源；无证据时必须输出“仍需补充证据”。
- 严禁泄露敏感信息（AK/SK/token/连接串）。

【输出规范】
- 每个阶段输出：设计决策、文件改动清单、回退策略、验收标准。
- Prompt 与实现都要显式体现“为何选择该工具/该上下文”。
```

## 3) 外部依赖识别增强 Prompt（重点）

```text
请实现“外部依赖识别引擎”，并明确区分：

一、RootSeeker 当前做法（现状）
- 读取构建文件作为证据输入 AI；
- 服务图/调用图侧重内部调用关系；
- 外部库依赖尚未统一结构化输出。

二、Cursor/Cline 可借鉴做法（目标）
- 先静态解析依赖声明，再按需动态解析实际依赖树；
- 对 Maven/Gradle 输出做标准化清洗，减少 AI 解析成本；
- 将依赖信息作为“可查询上下文”，支持后续多轮复用。

三、v3.0.0 目标实现
1) 新增工具 `deps.parse_external`
   - 输入: `project_root`, `manifest_file`(可选)
   - 支持: `pom.xml`, `build.gradle`, `build.gradle.kts`, `requirements.txt`, `pyproject.toml`
   - 输出:
     {
       "ecosystem": "maven|gradle|python",
       "direct_dependencies": [...],
       "declared_variables": [...],
       "risk_flags": ["version_variable", "dynamic_range", ...]
     }

2) 新增工具 `cmd.run_build_analysis`
   - 输入: `project_root`, `tool`, `command_type`
   - command_type 枚举:
     - maven: dependency_tree
     - gradle: dependencies
     - python: pip_freeze
   - 输出:
     {
       "resolved_dependencies": [...],
       "raw_output_excerpt": "...",
       "is_truncated": true|false
     }

3) 新增工具 `deps.diff_declared_vs_resolved`
   - 输入: `declared`, `resolved`
   - 输出: 漂移项（声明有但未解析、解析有但未声明、版本不一致）

四、触发策略（必须写入 Prompt）
- 默认先调用 `deps.parse_external`；
- 若命中以下条件才允许调用 `cmd.run_build_analysis`：
  - 版本变量无法解析（如 `${spring.version}`）；
  - 存在父 pom / BOM / Gradle 多配置导致静态不确定；
  - 问题类型明确与运行态依赖冲突相关（如 ClassNotFound/NoSuchMethodError）。
```

## 3.1 依赖代码识别增强 Prompt（JAR / site-packages）

```text
请实现“依赖代码识别”能力，目标是在分析 ClassNotFound/NoSuchMethodError/版本冲突时，AI 能像 IDE 一样定位到依赖库的定义位置与源码片段。

一、核心思想
- AI 不应假设能直接理解 jar/wheel 二进制；必须先把依赖解析为结构化信息（坐标、路径、符号索引或 sources）。
- 默认优先语义定位（Headless LSP）；当 LSP 不可用或返回不可读位置时，再回退到“源码 sources.jar / .py 文件”兜底，避免反编译 class。

二、Java（方案B：Headless JDT LS 优先，sources.jar 兜底）
1) 解析依赖坐标：通过 `deps.parse_external`（声明）或 `cmd.run_build_analysis`（解析后依赖树）拿到 groupId/artifactId/version。
2) 首选语义定位：启动 headless JDT LS，通过 LSP 能力获得 `definition/references/workspaceSymbols`，实现 Cursor 风格的“跳转定义/找引用/符号搜索”。
3) sources 兜底：当 JDT LS 不可用或初始化失败时，再走 `deps.fetch_java_sources` -> 物化源码 -> 索引 -> `code.resolve_symbol` 的退化路径。

三、Python（方案B：Headless Pyright/Pylsp 优先，site-packages 源码兜底）
1) 解析依赖与安装位置：
   - 优先从项目 venv：通过 `python -c` + importlib.metadata 获取包版本与安装路径
   - 无 venv 时才考虑系统 site-packages（必须可配置并显式允许）
2) 首选语义定位：启动 headless Pyright（或 python-lsp-server），通过 LSP 获得 `definition/references` 等语义信息。
3) 源码兜底：在 LSP 不可用时，回退为“site-packages 源码索引 + import 映射”。

四、触发策略（何时必须识别依赖代码）
- 错误签名属于依赖冲突（ClassNotFound/NoSuchMethodError/ImportError/ModuleNotFoundError）
- 或已定位到调用点但需要确认“依赖库方法签名/版本差异”
- 或需要对比 declared vs resolved 漂移项时
```

## 3.2 Headless LSP 方案B Prompt（Cursor 风格语义能力）

```text
请实现 Headless LSP（无 VS Code UI 环境）能力，并封装为 MCP 工具，让 AI 能像 Cursor 一样用“语义跳转”理解依赖库代码。

一、总体目标
- 通过 LSP 提供的语义接口，解决“依赖库代码无法定位/无法跳转/只能猜”的问题。
- 覆盖最小可用能力：workspace symbol、go to definition、find references、hover、document symbols。
- 支持 Java（JDT LS）与 Python（Pyright 或 python-lsp-server）。

二、工具清单（必须实现）
1) `lsp.start`
- 输入：language("java"|"python"), project_root, workspace_name(可选), extra(可选)
- 行为：启动对应 language server（stdio），完成 initialize/initialized，建立会话缓存（按 project_root+language）。若该会话已存在，则返回“已启动”并复用已有进程（幂等）。

2) `lsp.stop`
- 输入：language, project_root
- 行为：关闭会话并清理子进程。

3) `lsp.workspace_symbol`
- 输入：language, project_root, query, limit(默认 20)
- 输出：[{name, kind, containerName, location:{file_path, range}}]

4) `lsp.definition`
- 输入：language, project_root, file_path, line, character
- 输出：[{location:{file_path, range}, preview}]

5) `lsp.references`
- 输入：language, project_root, file_path, line, character, includeDeclaration(默认 false), limit(默认 50)
- 输出：[{location:{file_path, range}, preview}]

6) `lsp.hover`（可选但推荐）
- 输入：language, project_root, file_path, line, character
- 输出：{markdown, range}

7) `lsp.document_symbols`（可选但推荐）
- 输入：language, project_root, file_path
- 输出：[{name, kind, range, selectionRange, children?}]

三、会话与性能（必须实现）
- LSP 会话复用：同一 project_root+language 复用一个进程，避免每次 tool_call 都重新初始化。
- 超时与降级：initialize 或任一请求超时必须返回 TOOL_TIMEOUT，并标注 degraded_modes（例如 "lsp_unavailable"），供下一轮决策回退到 sources/纯搜索。
- 工作区隔离：默认只允许访问 project_root 内文件；额外允许访问“依赖缓存目录白名单”（例如 ~/.m2、~/.gradle、venv/site-packages），但必须显式配置并只读访问。

四、协议与坐标约定（必须写清）
- LSP 的 line/character 为 0-based；工具输入需明确使用 0-based（或在工具层做 1-based -> 0-based 适配，但必须统一）。
- LSP 的 location 使用 URI；工具输出需将 URI 转换为 `file_path`（相对 project_root）+ range。
- 若返回非 file URI：
  - jdt://：优先尝试将符号映射到依赖缓存目录白名单中的 sources（~/.m2、~/.gradle 等）并读取；若仍不可读或未命中 sources，再触发“依赖源码兜底”（`deps.fetch_java_sources` -> 物化到受控缓存目录 -> `code.resolve_symbol` 或 `code.read`）。
  - zip:// 等：优先在白名单目录内解析与读取；无法读取时再物化到受控缓存目录后读取。

五、最小协议流程（实现细则）
- `lsp.start` 需完成：
  - 启动子进程（stdio），建立 JSON-RPC 2.0 收发（Content-Length framing）
  - 发送 `initialize`（包含 rootUri/workspaceFolders/capabilities）
  - 等待 `initialize` 响应后发送 `initialized`
- 在调用 `lsp.definition/references/hover/document_symbols` 前：
  - 若 server 需要打开文档才能提供语义，工具层必须自动发送 `textDocument/didOpen`（读取 file_path 内容）并维护 version。

六、工程约束与运行方式（必须落地到部署）
- 语言服务器属于外部可执行依赖，必须明确获取与版本固定策略，避免线上环境不可用。
- 运行方式必须二选一（或同时支持，但必须有优先级）：
  - 直接运行（非 Docker）：
    - 在启动脚本（sh）中先启动 RootSeeker，再由 RootSeeker 内部按需启动 language server；或在启动脚本中提前启动 language server 并由 RootSeeker 连接/复用。
    - `lsp.start` 必须幂等：若进程已存在则提示“已启动”，不可重复拉起导致端口/资源冲突。
  - Docker 运行：
    - 必须将 JDT LS / Pyright（或 python-lsp-server）内置进镜像，且版本固定（镜像 tag 即版本）。
    - 禁止运行时临时联网下载二进制（除非明确允许且有缓存/校验）。

七、`lsp.start.extra` 最小字段定义（必须固定 Schema）
- Java（jdtls）至少需要：
  - java_home（可选，缺省则使用环境变量 JAVA_HOME 或系统 java）
  - jdtls_launcher_path（必填：org.eclipse.equinox.launcher_*.jar 的路径）
  - jdtls_config_dir（必填：jdtls 的 config_* 目录）
  - workspace_data_dir（必填：jdtls workspace 数据目录，必须在 RootSeeker 可写目录下）
  - log_level（可选：INFO/WARN/DEBUG）
- Python（pyright/pylsp）至少需要二选一：
  - python_path（显式解释器路径）
  - venv_dir（项目 venv 路径，由服务端推导 python_path）

八、Java（JDT LS）落地要点
- 需要 JDK 可用；必要时通过配置传入 JAVA_HOME。
- project_root 下若有 pom.xml/build.gradle，JDT LS 应自动导入并解析依赖；导入耗时必须可观测并具备超时控制。
- 若依赖缺少源码：允许配套调用 `deps.fetch_java_sources`（或执行 dependency:sources）后重试 definition。

九、Python（Pyright）落地要点
- project_root 下若有 venv，优先绑定该解释器；否则按配置允许 fallback 到系统 python。
- 对 site-packages 的解析以 LSP 为准；当 LSP 不可用时才做源码索引兜底。

十、Prompt 侧强约束（必须写入 Plan 提示词）
- 当需要定位依赖库符号（不在仓库内）时，优先使用 lsp.definition / lsp.workspace_symbol。
- 若 lsp 返回不可用或超时，再回退到 sources 兜底（deps.fetch_java_sources + code.search + code.resolve_symbol）。
```

## 4) AI 代码解析与上下文管理优化 Prompt（参考 Cursor/Trae/Cline）

> **Cline 参考**：工具错误处理已基于 cline-3.72.0 源码核对（见对比分析 B 项）。

```text
请在现有 Plan -> Act -> Synthesize -> Check 基础上，升级 AI 对代码解析能力：

1) 上下文分层管理（借鉴 Cursor/Trae）
- Layer A: GoalContext（问题目标、轮次状态、约束）
- Layer B: EvidenceContext（工具结果摘要、关键参数、失败重试线索）
- Layer C: CodeContext（符号定义、调用链、代码片段）
- 默认仅注入 A+B；C 按需懒加载，避免窗口污染。

2) 解析路径标准化（借鉴 Cursor/Cline）
- 统一顺序：
  1. 语义检索（code.search）
  2. 精确检索（关键词/正则）
  3. 依赖图约束（deps.get_graph）
  4. 代码读取（code.read）
  5. 综合分析（analysis.synthesize）
- 禁止一上来直接 full analysis。

3) Prompt 约束增强
- 每个步骤必须输出 `why`，且 why 必须体现“勘探 -> 定位 -> 分析”。
- 若证据不足，必须输出 `NEED_MORE_EVIDENCE`，并给出下一轮工具计划。
- 对每轮工具结果做相关性重排：优先保留与错误签名、trace_id、目标符号直接相关的证据。

4) 代码解析能力增强目标
- Java/Python 保持 AST 优先；
- 对非 AST 强语言采用“符号索引 + 约束检索 + 片段验证”兜底策略；
- 输出中必须区分“确定结论 / 高概率假设 / 待验证项”。
```

5) 工具错误与参数修正（借鉴 Cline 3.72.0）
- 统一错误格式：使用 <error> 标签包裹，便于 LLM 解析；缺参时明确标出 paramName（参考 Cline missingToolParameterError）。
- 渐进式错误：同一工具连续失败时，按 failure_count 分级提示（第 1/2/3+ 次逐步加强，参考 Cline writeToFileMissingContentError）。
- 参数修正策略：RootSeeker 采用「错误判断 AI」子流程输出 corrected_args 后重试；Cline 则依赖主 LLM 在下一轮看到 tool_result 后自行修正。两者均无显式「错误信息↔参数」映射，依赖 LLM 语义理解。
- mistake_limit：连续失败达上限时中止或回退，避免无限重试。
```

## 6) 对比分析 Prompt（必须先执行）

```text
在开始实现前，请先输出对比分析报告：

## 8) 实现路径（写入提示词计划的落地指引）

本节把 v3.0.0 的“实现路径”细化到 RootSeeker 现有模块边界，目标是：参考 Cursor/Trae/Cline 的思路来纠正当前实现的薄弱点，而不是只新增功能。

### 8.1 外部依赖识别（库依赖）

目标：把“构建文件作为证据”升级为“结构化依赖画像 + 必要时动态解析”。

- 新增 MCP 工具文件
  - `root_seeker/mcp/tools/cmd.py`
    - 新增工具：`cmd.run_build_analysis`
    - 只允许白名单 command_type -> argv 映射，严禁自由拼接字符串与 shell=True。
  - `root_seeker/mcp/tools/lsp.py`
    - 新增工具：`lsp.start`、`lsp.stop`、`lsp.workspace_symbol`、`lsp.definition`、`lsp.references`、`lsp.hover`、`lsp.document_symbols`
    - 目标：在无 IDE 环境下获得 Cursor 风格的语义跳转能力（依赖库/工程内符号都适用）
  - `root_seeker/mcp/tools/code.py`
    - 新增工具：`code.resolve_symbol`
    - 目标：当 LSP 不可用或返回不可读 URI 时，基于已物化源码（sources/site-packages）与仓库源码做符号兜底定位
  - `root_seeker/mcp/tools/deps_external.py`（或合并进现有 deps.py，但建议拆分）
    - 新增工具：`deps.parse_external`、`deps.diff_declared_vs_resolved`
    - 可选新增：`deps.scan_binaries`（识别本地 jar/whl/so/dylib 等“二进制依赖”作为额外证据）
  - `root_seeker/mcp/tools/deps_sources.py`
    - 新增工具：`deps.fetch_java_sources`、`deps.index_dependency_sources`
    - 目标：把依赖坐标映射到可读源码（sources.jar / site-packages），并提供可搜索/可读取的索引入口

- 新增服务层（复用到 HTTP 与 MCP 两端）
  - `root_seeker/services/external_deps.py`
    - `parse_maven_pom(project_root, pom_path=None) -> DeclaredDeps`
    - `parse_gradle_build(project_root, build_file=None) -> DeclaredDeps`
    - `parse_python_manifest(project_root) -> DeclaredDeps`
    - `diff_declared_vs_resolved(declared, resolved) -> DriftReport`
    - `scan_binaries(project_root, patterns=["*.jar","*.whl","*.so","*.dylib"]) -> BinaryDeps`
  - `root_seeker/services/dependency_sources.py`
    - `fetch_java_sources(project_root) -> list[DepCoordinate]`
    - `materialize_maven_sources(coords) -> list[MaterializedSourceRoot]`
    - `index_source_roots(source_roots) -> IndexRef`
    - `resolve_symbol_in_sources(symbol, index_ref) -> list[CodeLocation]`
  - `root_seeker/services/lsp_manager.py`
    - `start(language, project_root) -> SessionRef`
    - `stop(language, project_root) -> None`
    - `workspace_symbol(session, query, limit) -> list[SymbolLocation]`
    - `definition(session, file_path, line, character) -> list[CodeLocation]`
    - `references(session, file_path, line, character, include_declaration) -> list[CodeLocation]`
    - `hover(session, file_path, line, character) -> HoverInfo`

- 解析策略（写进 Prompt，确保 AI 行为一致）
  - 默认走静态：`deps.parse_external`
  - 仅在触发条件命中时才走动态：`cmd.run_build_analysis`
  - 动态输出必须“标准化清洗”：只保留依赖坐标/版本/冲突提示 + 原始输出节选（可截断）
  - 依赖代码定位优先走 LSP：`lsp.workspace_symbol` / `lsp.definition` / `lsp.references`
  - 仅在触发条件命中且 LSP 不可用时才走依赖源码：`deps.fetch_java_sources` / `deps.index_dependency_sources` / `code.resolve_symbol`

### 8.2 上下文管理与代码解析（IDE 能力对齐）

目标：把 RootSeeker 现有“多轮编排”升级为更像 Cursor/Trae/Cline 的 IDE 级上下文策略，并修复当前实现中会导致“悄悄降级”的问题。

- 需要修改的关键位置
  - `root_seeker/ai/orchestrator.py`
    - 将关键 `except Exception: pass` 改为“结构化降级 + 可观测信号”（至少 warning + analysis_id + tool_name）
    - 将“字符截断”升级为“结构感知截断”
      - 对 JSON 类 tool_result：保留关键字段（file_path/line/hits 等）+ 截断 preview
      - 在 tool_result 文本中附加 `is_truncated=true` 与 `original_length`
    - 将“只保留最近 N 条”的压缩升级为“相关性保留”
      - 保留：首次定位证据 + 最近证据 + 每轮摘要索引（hash->引用）
  - `root_seeker/ai/context_discovery.py`
    - 让引用发现不再只看前 2000 字符：改为“抽样窗口 + 关键行优先”（stacktrace/traceId 行优先）
    - 输出必须携带“发现覆盖率/截断提示”，避免 Plan 误判“没有 trace_id”
  - `root_seeker/services/service_graph.py`
    - 扩展依赖线索：`lb://service`、Feign 的 `value/url`、占位符 `${...}`（先保守标记为低置信度）
    - 对扫描上限（前 5000 行、文件类型集合）与读取失败提供可解释元信息（避免静默缺边）
  - `root_seeker/services/call_graph_expander.py`
    - Tree-sitter 失败/Zoekt 失败不再静默：记录一次“降级事件”，并在返回的 evidence notes 中标注
    - 扫描上限（如 max_scan_files=200）触发时必须可见：返回 `risk_flags: ["scan_truncated"]`

- 上下文分层（写入 Prompt 并要求输出）
  - GoalContext：目标、轮次、约束、优先级
  - EvidenceContext：工具结果摘要、可复现参数、失败重试线索
  - CodeContext：符号定义、调用链、代码片段（按需懒加载）

## 9) 已知实现问题（用于 v3.0.0 纠偏优化）

本节要求在 v3.0.0 中“明确修复或显式降级”，避免继续出现“看起来能跑但结果变差”的隐性问题（参考 Cline 的可解释降级思路）。

```text
请基于当前实现的实际问题，设计 v3.0.0 的纠偏优化方案，并写明：

参考现有风险点（必须核对并修复/显式降级）：
- Orchestrator 多处 JSON 反解析失败会静默吞掉，导致 evidence_ctx/报告提取悄悄失效
- Context discovery 只取 error_log 前 2000 字会漏检 trace_id/栈关键信号
- Service graph/Call graph 多处文件读取与解析失败会静默 pass，导致缺边但不可解释
- Call graph 扫描上限触发不可见，可能把“没扫到”误当成“没有调用”

1) 静默异常问题（必须处理）
- 任何关键链路出现 `except Exception: pass` 都必须改为“可观测降级”，至少记录：analysis_id、tool_name、exception_class。

2) 截断与压缩问题（必须处理）
- 统一截断策略必须让 LLM 可感知（is_truncated/original_length）。
- 压缩不能只保留“最近 N 个”，必须保留“首次定位证据 + 最近证据 + 轮次摘要索引”。

3) 依赖图漏检问题（必须处理）
- service_graph 的规则扩展（lb://、Feign 参数变体、占位符）至少提供低置信度边，并让报告中可解释。
- call_graph 的 tree-sitter/Zoekt 失败必须在证据中可见，避免把“解析失败”当作“无调用”。

4) 上限触发不可见问题（必须处理）
- 任何扫描上限触发（如 5000 行、200 文件）都必须在输出里体现 risk_flags，供后续决策。
```
