# v2.0.0 提示词计划 vs 当前实现差异

## 一、流程结构对比

| 阶段 | v2.0.0 计划 | 当前实现 | 差异 |
|------|-------------|----------|------|
| Plan | 输出 JSON 计划，含 goal + steps，每步有 why | ✅ 已实现 | 一致 |
| Act | 逐步执行 tool calls，参数校验、结果压缩 | ✅ 已实现 | 计划要求「结果压缩与抽取」「保留可复现查询参数」，当前仅做截断 |
| Synthesize | 把 tool 结果转成 Evidence/Enrichment，LLM 生成报告 | ✅ 已实现 | 计划强调「转成 Evidence」，当前直接拼 tool_results 文本给 LLM |
| Check | 覆盖性、一致性、可复现性、安全性；可追加 0~2 次 tool calls | ⚠️ 部分实现 | 见下文 |

## 二、Plan 提示词差异（核心）

### v2.0.0 计划（第 5 节 244-264 行）

```json
{
  "goal": "定位 OrderService 创建订单失败的根因",
  "steps": [
    {"tool_name": "code.search", "args": {...}, "why": "让我先搜索 createOrder 方法的定义，确认相关代码位置"},
    {"tool_name": "code.read", "args": {...}, "why": "找到文件了，让我读取具体的代码实现，查看逻辑漏洞"},
    {"tool_name": "analysis.run", "args": {...}, "why": "结合代码和错误日志进行深度分析"}
  ]
}
```

- **思路**：code.search → code.read → analysis.run，强调「勘探 -> 定位 -> 分析」
- **analysis.run**：作为**最后一步**，在已有 code 证据基础上做深度分析

### 当前实现（prompts.py AI_ORCHESTRATOR_PLAN_USER）

```json
{
  "steps": [
    {"tool_name": "index.get_status", ...},
    {"tool_name": "correlation.get_info", ...},
    {"tool_name": "code.search", ...},
    {"tool_name": "analysis.run", "why": "执行完整分析"}
  ]
}
```

- **约束**：`优先使用 index.get_status、correlation.get_info 获取上下文，再用 code.search/code.read 勘探，最后用 analysis.run 完成分析`
- **问题**：示例和约束都**明确引导** LLM 把 `analysis.run` 作为必选步骤

### 差异总结

| 项目 | v2.0.0 计划 | 当前实现 |
|------|-------------|----------|
| 勘探顺序 | code.search → code.read → analysis.run | index.get_status → correlation.get_info → code.search → analysis.run |
| analysis.run 定位 | 最后一步，结合已有证据做深度分析 | 作为「完成分析」的默认步骤，常被优先调用 |
| 纯 Synthesize 路径 | 可用 code.search + code.read 结果直接 Synthesize | 未强调；Plan 倾向包含 analysis.run |
| 显式思考链 | 要求「模拟 IDE 开发者」「描述勘探思路」 | 有 why 字段，但未强制「勘探 -> 定位 -> 分析」的显式思考 |

## 三、执行流程差异（实际行为）

### v2.0.0 计划预期

1. Plan 生成：code.search → code.read → (可选) analysis.run
2. Act 执行细粒度工具，收集代码证据
3. Synthesize 基于 tool 结果生成报告（或由 analysis.run 完成）
4. 不确定时优先 index.get_status、correlation.get_info、deps.get_graph

### 当前实际行为（从日志推断）

1. Plan 生成：常包含 analysis.run 作为早期或唯一分析步骤
2. Act 执行：第一步或第二步就调用 analysis.run
3. **analysis.run 内部**：`skip_ai_driven=True`，走直连路径（enricher → zoekt → vector → evidence → 多轮 LLM）
4. 结果：AI 驱动只负责 Plan，**实际分析仍是传统多轮对话**

### 根本原因

- Plan 提示词把 `analysis.run` 写进示例和约束，LLM 倾向于直接调用
- 未明确「优先用 code.search/code.read 勘探，仅在证据不足时再用 analysis.run」
- 未提供「纯 Synthesize」路径的明确引导（只用细粒度工具 + Synthesize，不调用 analysis.run）

## 四、Check 阶段差异

### v2.0.0 计划（第 5 节 261-265 行）

- **覆盖性**：错误签名、关键日志证据、repo_id/服务名、时间窗、correlation_id
- **一致性**：结论与证据一致；无矛盾证据；无未经证据支撑的断言
- **可复现性**：复现所需查询参数（query_key/时间窗/trace_id/request_id）及缺失项清单
- **安全性**：脱敏 AK/SK/token/连接串
- **追加 tool calls**：最多 0~2 次，且必须说明「追加原因」和「预期补齐字段」

### 当前实现（orchestrator._check_and_sanitize）

- **覆盖性**：仅检查 service_name、summary/hypotheses/suggestions 非空
- **一致性**：未实现（无证据与结论的交叉校验）
- **可复现性**：未实现（无 query_key、时间窗、trace_id 等检查）
- **安全性**：✅ 已实现 redact_sensitive 脱敏
- **追加 tool calls**：needs_extra 时固定调用 correlation.get_info，**未要求说明追加原因和预期补齐字段**

## 五、Act 阶段差异

### v2.0.0 计划（第 5 节 256-259 行）

- 每次调用前：参数校验与默认值补全（时间窗、repo_id、correlation_id）
- 每次调用后：**结果压缩与抽取**（仅保留证据字段与可追溯引用）
- 返回过大：截断并**生成摘要**，**保留可复现查询参数**

### 当前实现

- 参数补全：✅ `_fill_step_args` 补全 service_name、query_key、error_log、error_event、analysis_id、deps.get_graph 的 target
- 结果处理：仅 `_truncate_text` 截断，**无结果压缩、无摘要生成、无保留可复现查询参数**

## 六、可靠性策略差异

### v2.0.0 计划（第 5 节 267-270 行）

- 对不确定信息（repo_id、时间窗）：**优先** index.get_status 或 correlation.get_info，**而不是猜测**
- 需要定位接口影响面/调用链：**优先** deps.get_graph，再决定检索/分析范围，**而不是全量搜索**
- tool errors：归一化为统一错误结构，便于 LLM 决策重试/换工具/回退

### 当前实现

- 参数补全：从 event 推断，未强制「不确定时先调 index.get_status/correlation.get_info」
- deps.get_graph：Plan 示例未包含，约束也未强调「定位调用链时优先 deps.get_graph」
- tool errors：✅ 已实现 _call_tool_with_retry，错误判断 AI 修正参数后重试

## 八、优先修复建议（按影响排序）

1. **Plan 提示词**：弱化 analysis.run，增加「纯 Synthesize」路径示例，引导 LLM 优先用 code.search、code.read、deps.get_graph 勘探，仅在必要时再用 analysis.run。
2. **Check 一致性**：若 report 与 tool_results 明显矛盾，可标记 needs_extra 或要求重新 Synthesize。
3. **Check 追加逻辑**：当前固定 correlation.get_info；可改为由 Check 阶段 LLM 决定追加哪个工具及原因（若实现成本可接受）。
