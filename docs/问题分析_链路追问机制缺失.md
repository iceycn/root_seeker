# 问题分析：为何未继续追问「sendOAMessageUsers 为何为空」

## 一、现象

分析结论：
- **快速定位**：发送课程审批 OA 消息时发生 RPC 接口错误
- **根本原因**：sendCourseApprovalOAMessage 方法中 sendOAMessageUsers 集合为空，导致抛出 BusinessException。本质是业务逻辑错误（上游数据缺失或处理逻辑缺陷），而非真正的 RPC 接口错误。

**用户期望**：应继续分析链路，追问「为什么 sendOAMessageUsers 为空？」——该集合从哪来？上游是谁？配置/数据源/调用方是否有问题？

**实际行为**：分析在此停止，未触发下一轮证据收集。

---

## 二、根因排查

### 2.1 RootSeeker 的「继续分析」触发机制

| 机制 | 触发条件 | 实现位置 |
|------|----------|----------|
| **NEED_MORE_EVIDENCE** | Synthesize 的 LLM 输出中显式包含 `NEED_MORE_EVIDENCE` 数组 | `orchestrator.py` 781–798 行；`_collect_evidence_recursive` |
| **Check 下一轮决策** | `_decide_next_round` 返回 `continue_analysis: true` | `orchestrator.py` 815–837 行 |

### 2.2 为何本次未触发 NEED_MORE_EVIDENCE？

1. **Prompt 语义**：`COMMON_ERROR_PATTERNS_HINT` 中 NEED_MORE_EVIDENCE 的触发条件是：
   > 「若对某处不清楚、**证据不足以确定根因**，请明确输出 NEED_MORE_EVIDENCE」

2. **LLM 的隐含判断**：LLM 认为「sendOAMessageUsers 集合为空」就是根因，因此**没有**输出 NEED_MORE_EVIDENCE。它没有意识到「集合为空」本身需要进一步追问「为什么为空」。

3. **Prompt 已有但未强化的规则**：

   ```
   2. 空值/空指针：NPE、Optional 未校验、集合为空时直接 get(0)。
      检查：上游返回值、外部输入、配置项是否可能为 null/空。
   ```

   这里只要求「检查」上游，但未明确要求：**若发现集合为空/数据缺失，应输出 NEED_MORE_EVIDENCE 继续追溯上游**。

### 2.3 为何 Check 下一轮也未触发？

- Check 阶段在 Synthesize 之后执行，输入是「本轮回告摘要 + 工具结果摘要」。
- 若 Synthesize 已给出「结论」（集合为空），Check 的 LLM 可能判断「结论已可靠」或「证据充分」，从而 `continue_analysis: false`。
- Check 的 prompt 未显式强调「若根因涉及数据缺失/集合为空，应继续追问上游链路」。

## 三、Cline 对比

### 3.1 Cline 的「继续」机制

| Cline 设计 | 说明 |
|------------|------|
| **tool_use 循环** | 每次 API 请求后，若有 tool_use 块，继续执行工具并将结果回写 conversation，再请求 API，直到无新 tool_use |
| **focus chain** | 任务进度清单（获取上下文→定位代码→收集证据→分析根因），注入 Plan 提示，帮助模型规划步骤 |
| **mistake limit** | 同一工具连续失败 N 次时停止，避免死循环 |

Cline 的「继续」是**工具驱动**：模型每次可决定是否再调用工具。

### 3.2 RootSeeker 与 Cline 的差异

| 维度 | RootSeeker | Cline |
|------|------------|-------|
| 继续条件 | Synthesize 输出 NEED_MORE_EVIDENCE，或 Check 输出 continue_analysis | 每次有 tool_use 就继续 |
| 追问粒度 | 依赖 Synthesize/Check 的 LLM 主动输出 | 模型可随时决定「再读一个文件」「再搜一次」 |
| 中间结论 | 无显式「中间结论→继续追问」规则 | 无 explicit 规则，但模型可自然延续对话 |

### 3.3 Cline 是否有「发现新问题后继续追问」的显式机制？

- **无**：Cline 没有显式 prompt 规则「若发现集合为空，必须继续追问上游」。
- **有**：Cline 的 tool_use 循环天然支持「模型发现需要更多信息时，再发 tool_use」。若模型在分析过程中意识到「需要看 sendOAMessageUsers 从哪来」，可以再发 code.search / code.read。
- **差异**：RootSeeker 的 Synthesize 是「一次性总结」，模型在总结时若未输出 NEED_MORE_EVIDENCE，流程就结束；Cline 的对话是连续的，模型可在多轮中逐步深入。

## 四、建议（不改代码，仅排查结论）

### 4.1 根因总结

| 问题 | 说明 |
|------|------|
| 根本原因 | Synthesize 的 LLM 将「集合为空」视为最终根因，未输出 NEED_MORE_EVIDENCE，导致未触发下一轮证据收集 |
| 直接原因 | Prompt 中「集合为空/数据缺失时应继续追问上游」的规则不够显式，未强制要求输出 NEED_MORE_EVIDENCE |
| 机制差异 | RootSeeker 依赖 Synthesize 一次性输出；Cline 依赖多轮 tool_use 自然延续 |

### 4.2 可借鉴的 Cline 思路

1. **显式追问规则**：在 Synthesize 的 prompt 中增加：若根因涉及「集合为空」「数据缺失」「上游返回值异常」，应输出 NEED_MORE_EVIDENCE，列出需追溯的上游（如：sendOAMessageUsers 的赋值来源、调用方、配置项）。
2. **Check 强化**：在 Check 的 prompt 中增加：若 summary 含「集合为空」「数据缺失」等表述，应倾向于 continue_analysis=true，并给出 next_round_evidence_needs（如：sendOAMessageUsers 的赋值逻辑、调用链上游）。
3. **focus chain 扩展**：在「分析根因」后增加「追溯上游链路」步骤，当发现数据缺失时引导模型进入该步骤。

### 4.3 建议的验证方式

1. 查看该 analysis 的 audit 日志，确认 Synthesize 的原始 JSON 是否包含 NEED_MORE_EVIDENCE。
2. 若确认无，则根因是 LLM 未输出；若包含但未触发，则需检查 orchestrator 的解析逻辑。

## 五、已实施优化（参考 Cline）

1. **COMMON_ERROR_PATTERNS_HINT**：新增【链路追问】规则，若根因涉及「集合为空」「数据缺失」「上游返回值异常」，必须输出 NEED_MORE_EVIDENCE 继续追溯上游。
2. **AI_ORCHESTRATOR_SYNTHESIZE_USER**：增加 NEED_MORE_EVIDENCE 字段说明与【链路追问】约束。
3. **AI_ORCHESTRATOR_NEXT_ROUND_***：Check 阶段增加「若 summary 含集合为空/数据缺失，应 continue_analysis=true」规则。
4. **build_focus_chain**：增加「追溯上游链路」步骤，引导后续轮进入该阶段。
5. **orchestrator._report_from_parsed**：解析 NEED_MORE_EVIDENCE 并写入 report。
6. **orchestrator 主循环**：_synthesize 返回 need_more_evidence 时触发 _collect_evidence_recursive 收集证据。
