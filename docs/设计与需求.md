# 公司级 AI 自动代码分析工具：设计与需求

> 本文档结合当前 RootSeeker 项目，对「公司级 AI 自动代码分析工具」的设计背景、要求、选型与实施做统一整理；并给出项目结构、文档整合与优化结论。

---

## 一、角色与背景

**角色**：Python 专家，设计公司级别使用的 AI 自动代码分析工具。

**背景**：需要一套系统，在接收到错误信息后：

1. 根据错误信息中的关键信息（如**服务名**）路由到指定仓库；
2. 在指定仓库中通过 AI 检索内容；
3. 分析代码片段，进一步找出更多的关联内容、上下游方法片段，直到找出所有的关联内容；
4. 分析错误原因并给出修改建议；
5. 列出关联服务。
6. 支持通过关联项在多个仓库中检索关联代码片段
7. ✅ **已实现**：支持通过 `trace_id`/`request_id` 自动查询日志链，获取完整的调用链上下文
8. ✅ **已实现**：完善的日志系统，可配置日志级别，关键流程有详细日志记录

**输出风格**：偏问题排查与定位。

## 三、思考与选型（已落地部分）

### 1. 代码向量化与快速检索

- **向量库**：Qdrant（生产可用、过滤强、Python 生态成熟）。已定义 `VectorStoreProtocol`（`root_seeker/protocols.py`），后续可接入 Milvus、pgvector 等。
- **切分**：Tree-sitter 按函数/类/块切分，利于代码检索。已定义 `ChunkerProtocol`，当前 TreeSitterChunker 符合，后续可替换为其他切分策略。
- **向量化**：FastEmbed 本地向量化（可换为其他 OpenAI 兼容 embedder）；支持 hash 降级。
- **词法检索**：Zoekt（trigram/regex，适合大仓），与向量检索混合使用。已定义 `LexicalSearchProtocol`，ZoektClient 符合，后续可接入 OpenGrok 等。

### 2. 类似 Trae/Cursor 的快速代码分析

- 本方案采用「错误堆栈 + 词法/向量检索 + 证据包 + LLM」流水线，实现「根据错误堆栈快速定位代码与原因」。
- 堆栈解析：从 Java/Python 堆栈提取 `(文件:行号)`，直接读本地代码片段作为高优先级证据（`evidence.py` 中 `_collect_evidence_from_stack_trace`）。
- **跨仓库关联证据**：`cross_repo_evidence: true` 时，在依赖图中关联服务（上下游）对应仓库内做向量检索，将命中片段合并进证据包（`EvidenceBuilder.append_vector_hits_from_repo`），支持「通过关联项在多个仓库中检索关联代码片段」。
- **多轮对话优化**：✅ **已实现**：支持三种 LLM 多轮对话模式（staged、self_refine、hybrid），默认启用混合模式，提升分析质量。详见 `LLM多轮对话实现.md`。
- 可选参考：Continue、Tabby 等自托管方案，本项目中以自研流水线为主。

#### 「分析代码片段，进一步找出关联内容、上下游方法片段」实现情况

| 维度 | 说明 | 当前实现 |
|------|------|----------|
| **服务级关联** | 从仓库代码解析「服务 A 调用了服务 B」，得到上下游服务列表 | ✅ 已实现：`service_graph` 从 HTTP/Feign 等解析 caller→callee 服务名；报告中有 `related_services`；可选 `cross_repo_evidence` 在关联服务仓库内做向量检索并合并证据。 |
| **同仓库内方法级上下游** | 从已收集的代码片段（如堆栈中的 `A.java:100`）解析「谁调用了 A」「A 调用了谁」等**方法级**调用关系，再取这些**调用方/被调用方方法**的代码片段加入证据，并可迭代直到上限或找全 | ✅ **已实现并优化**：`CallGraphExpander`（`services/call_graph_expander.py`）支持：<br>1. **Tree-sitter 精确解析**：优先使用 Tree-sitter 解析方法调用关系（`call_graph_use_tree_sitter: true`），回退到正则<br>2. **异步 Zoekt 集成**：支持异步调用 Zoekt 搜索方法名，优先使用 Zoekt 定位方法<br>3. **性能优化**：方法定位缓存（LRU）、文件解析缓存、扫描范围限制（`call_graph_scan_limit_dirs`，优先扫描 `src/main`、`src` 等常见目录）、全仓库扫描限制文件数（最多 200 个）<br>配置项：`call_graph_expansion`、`call_graph_max_rounds`、`call_graph_max_methods_per_round`、`call_graph_max_total_methods`、`call_graph_use_tree_sitter`、`call_graph_scan_limit_dirs`、`call_graph_cache_size`。 |
| **「直到找出所有的关联内容」** | 迭代扩展证据直到无新发现或达到上限 | ✅ **已实现**：`CallGraphExpander.expand_evidence` 支持迭代扩展：从初始证据中提取方法调用，定位并读取这些方法的代码，再从新加入的代码片段继续解析调用关系，迭代直到达到 `max_rounds`、`max_total_methods` 或 `max_files`/`max_chars_total` 上限，或本轮无新发现时停止。每轮扩展后会记录到 `evidence.notes`。 |

结论：**「分析代码片段，进一步找出更多的关联内容、上下游方法片段，直到找出所有的关联内容」** 已完整实现：**服务级关联 + 跨仓库检索**（`cross_repo_evidence`）+ **同仓库内方法级调用链展开与迭代扩展**（`call_graph_expansion`）。

### 4. 仓库同步与索引维护

- ✅ **已实现定时任务**：`PeriodicTaskService` 支持定期同步仓库并更新向量索引
  - **智能变更检测**：使用 `git rev-list --count HEAD..@{upstream}` 检测是否有代码变更
  - **增量更新**：只为有变更的仓库（`status="updated"` 或 `status="cloned"`）执行 pull 和索引更新
  - **配置灵活**：支持总开关、同步间隔、索引间隔、并发数等配置
  - **手动触发**：仍保留 `POST /repos/sync` 和 `POST /index/repo/{service_name}` 接口用于手动触发

### 3. 与业务打通：多来源错误输入与日志链查询

- **优先入口**：阿里云 SLS（日志服务）。Webhook 接收 JSON（含 `service_name`、`error_log`），再按 `query_key` 拼接 SQL 模板主动拉取更多日志。配置项 `ingest_source` 已预留（默认 `aliyun_sls`），后续可扩展 Sentry/Kafka 等并按配置选择。
- **查询 SQL 生命周期**：每条事件携带 `query_key`，工具用 `sql_templates` 中对应模板拼接 SQL（含 `{service_name}`、`{start_ts}`、`{end_ts}` 等）。
- **可扩展**：接入层为 IngestEvent → NormalizedErrorEvent；新增 Sentry/Kafka 等只需实现对应 Adapter 并写入同一事件模型。
- **日志链自动查询（Trace Chain）**：✅ **已实现并优化**：
  - **接口化设计**：定义了 `TraceChainProvider` Protocol 接口（`providers/trace_chain.py`），支持通过 `trace_id`/`request_id` 查询日志链
  - **智能提取**：优先从 `event.tags` 提取，其次使用 LLM 智能提取，最后回退到正则匹配
  - **实现方式**：
    - `EmptyTraceChainProvider`：未配置时返回空列表（避免异常）
    - `AliyunTraceChainProvider`：阿里云 SLS 实现，使用全文查询（不依赖字段名），自动处理特殊字符
  - **时间范围限制**：可配置的最大时间窗口（`max_trace_chain_time_window_seconds`，默认 300 秒/5 分钟），超过自动调整
  - **配置项**：`trace_chain_enabled`、`trace_chain_time_window_seconds`、`max_trace_chain_time_window_seconds`
  - **使用场景**：当启用调用链查询时，系统会自动提取 `trace_id`/`request_id`，查询相关日志并合并到证据包中，提供完整的调用链上下文

### 4. 对外输出

- **企业微信**：Markdown 推送至群（webhook_url）。
- **钉钉**：Markdown 推送至群（webhook_url）。
- **数据库/文件存储**：`report_store_path` 非空时，将 Markdown 报告追加写入该文件，便于持久化或后续导入数据库。
- **日志打印**：`notify_console: true` 时，将 Markdown 报告通过 logging 打印到控制台。
- 实现：`providers/notifiers.py`，支持多通道：WeCom、DingTalk、ConsoleNotifier、FileReportStoreNotifier；统一 Report 渲染后按配置的 notifiers 列表依次发送。

### 5. 可扩展的来源与日志源

- **阿里云 SLS**：已实现（AK/SK、project、logstore、sql_templates）；AK/SK 允许落盘于配置。
- **Trace Chain Provider**：已实现接口化设计（`TraceChainProvider` Protocol），当前支持：
  - 空实现（`EmptyTraceChainProvider`）：未配置时使用
  - 阿里云 SLS 实现（`AliyunTraceChainProvider`）：通过 `trace_id`/`request_id` 查询日志链
  - 后续可扩展其他实现（如 Sentry、Jaeger 等）只需实现 `TraceChainProvider` 接口
- **其他云/自建**：实现 `CloudLogProvider` 接口并注入即可，无需改核心流水线。

### 6. 定时任务：自动同步仓库并更新索引

- ✅ **已实现**：`PeriodicTaskService`（`runtime/periodic_tasks.py`）支持定时同步仓库并更新向量索引
- **功能特性**：
  - **总开关控制**：`periodic_tasks_enabled` 统一控制定时任务功能
  - **智能变更检测**：使用 `git rev-list --count HEAD..@{upstream}` 检测是否有代码变更，只有真正有变更的仓库才会执行 pull 和索引更新
  - **增量索引更新**：只为有变更的仓库（`status="updated"` 或 `status="cloned"`）触发向量索引更新，避免不必要的全量索引
  - **并发控制**：支持配置并发数（`auto_sync_concurrency`，默认 8），避免对 Git 服务器压力过大
  - **灵活配置**：
    - `auto_sync_enabled`：是否启用定时仓库同步（git pull）
    - `auto_sync_interval_seconds`：仓库同步间隔（默认 3600 秒/1 小时）
    - `auto_index_enabled`：是否启用定时向量索引更新
    - `auto_index_after_sync`：同步完成后是否自动触发索引（默认 true，仅对有变更的仓库）
    - `auto_index_interval_seconds`：向量索引更新间隔（默认 7200 秒/2 小时，仅在 `auto_index_after_sync=false` 时独立运行）
- **工作流程**：
  1. 定时任务启动（`periodic_tasks_enabled=true`）
  2. 每隔配置的间隔执行同步：对每个仓库执行 `git fetch` + 检查变更
  3. 检测变更：使用 `git rev-list` 或检查 `git pull` 输出判断是否有更新
  4. 状态分类：`updated`（有更新）、`no_change`（无变更）、`cloned`（新克隆）、`error`（失败）
  5. 自动索引：如果启用且 `auto_index_after_sync=true`，只为有变更的仓库更新向量索引
- **日志记录**：详细记录同步结果、变更统计、索引更新进度等，便于运维监控
